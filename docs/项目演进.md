> 目标一句话：
> 把 Beat Link Trigger（BLT）的音乐事件（player、phraseBeat、位置等）通过一条“意图层（intent）”安全地映射到 MA2（通过 bus-to-ma2 9940 + ma2 telnet 9930），同时不破坏既有 `rules.yml` 触发体系。

## 服务/端口总览（你现在在用的那套）

- **9910 blt-osc-ingest**：
  - 接 BLT/OSC/事件 → 规范化 → 提供 Web UI（过滤、播放器状态）
  - 提供 overlay 波形代理：`/overlay/wave-preview/:player`、`/overlay/wave-detail/:player`、`/overlay/params.json`
  - 产生合成事件：`trackChanged`（换歌）
  - “只监听 On-Air” 跟随逻辑（含 fallback 到 playing）

- **9960 intent-engine**：
  - 订阅 bus 事件（phraseBeat/trackChanged 等）
  - 产出通用、跨场地的 **intent**（scene/hit/energy/brightness/speed/movement/strobe/colorMood + beam/wash/spot/strobeFx layer）
  - DeepSeek AI 只参与 **意图生成/结构分析**，不直接控 MA2
  - Web UI：状态、调试 payload、AI 配置、AI chat、结构分析展示

- **9940 bus-to-ma2**：
  - 原有 `rules.yml`：传统规则触发
  - 新增 **parallel mode**：从 intent-engine 拉 intent → `intent-map.yml` 映射到 MA2 动作，不影响 rules
  - 提供 Web UI：Intent 开关/重载 intent-map（无需重启 Docker）

- **9930 ma2-telnet-remote**：
  - 负责和 MA2 Telnet 通信，把“执行动作”落在 MA2（executor/cue/pulse 等）

- **9950 control-console**：
  - 一个聚合控制台，把 9910/9930/9940 等操作集中（iframe 聚合、常用操作入口）

## 核心设计决策

### 1) Parallel mode（并行模式）
- **原因**：现场已经有大量 `rules.yml` 规则在跑，不能因为接入 intent 就破坏原触发链。
- **做法**：9940 内部保留 `rules.yml` 原逻辑；另起一条“intent 映射链”并行执行。
- **结果**：
  - 你可以随时 UI 一键开/关 intent，不影响老规则
  - intent-map 可快速迭代，不用动 rules.yml

### 2) Intent 作为通用协议层（跨场地/不依赖具体灯具）
- intent 不描述“某个灯具”，只描述“你想要的效果层”：
  - scene（INTRO/BUILD/DROP/BREAK/CHORUS/FILL/IDLE）
  - hit（瞬态打点）
  - energy/brightness/speed/movement/strobe/colorMood
  - layer（beam/wash/spot/strobeFx）
- 具体落地由场地侧 `intent-map.yml` 负责：
  - 同一份 intent，可以在不同场地映射到不同 executor/cue

### 3) AI 只做意图，不直接控 MA2（guardrail）
- AI（DeepSeek）在 9960 里：
  - 输出 scene/风格/建议
  - 永远不直接发 telnet
- 真正控制 MA2 的只有 9940（可审计、可开关、可回退）。

### 4) Token 策略：事件驱动（phraseBeat-only）
- 为了省 token：
  - AI 默认只在 `phraseBeat` 触发（且有冷却间隔）
  - 换歌 `trackChanged` 可触发一次关键分析

## 演进时间线（按提交里程碑）

> 下面是“从最初到现在”的主线，不列所有 UI 微调。

1. **初始套件**：BLT→OpenClaw + MA2 Telnet
   - 目标：先让链路跑通（事件能进来、MA2 能动作）。

2. **9940 bus-to-ma2**：从 WS bus → MA2 telnet，加入规则系统
   - `rules.yml`、cooldown、reload、基本 Web UI。

3. **9950 control-console**：统一控制台
   - 聚合 9910/9930/9940，减少来回切页面。

4. **9960 intent-engine**：引入 intent 层 + Web UI
   - 从“规则触发”升级为“意图驱动”。

5. **9940 parallel mode + intent-map.yml**
   - intent 驱动并行接入，不动 rules.yml。
   - UI 支持 intent toggle & reload intent-map。

6. **DeepSeek AI 接入 9960**
   - AI 配置 UI、/api/ai/config、/api/ai/test、run-once
   - AI chat + 调试 payload 可视化
   - phraseBeat-only + cooldown 降 token

7. **9910 waveform + on-air follow + trackChanged**
   - 9910 提供波形代理与静态全曲显示（不自动刷新）
   - “只监听 On-Air” + fallback 到 playing
   - trackChanged 合成事件（给 AI/intent 用）

8. **换歌结构分析（最新）**
   - 9960 在 trackChanged 时：
     - 拉 wave-preview PNG → 本地提取能量曲线（60 点）
     - 调 AI 输出：段落列表、DROP/BREAK 时间点（秒）、风格建议
   - 9960 UI 增加“本曲结构分析”展示。

## 当前建议的工作流（现场）

1) 先用 9940 只跑 `rules.yml`（稳定）
2) 打开 9940 Intent（parallel）测试 intent-map 的映射是否符合你分配的 Exec 角色
3) 9960 打开 AI：
   - 让它在 phraseBeat 低频介入（判段落/风格）
   - 换歌时自动做结构分析（DROP/BREAK 点 + 风格）
4) 根据结果迭代：
   - intent-map.yml（场地映射）
   - AI prompt / schema（稳定性）
