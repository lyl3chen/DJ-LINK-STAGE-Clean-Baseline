## 02:37 LTC 帧时基抖动修复实施

### 修改概述
基于历史相似问题的解决路径，对 `LtcDriver.java` 实施了帧锁定连续步进机制，替换原有的 wall-clock 式实时重贴。

### 修改文件
**`dbclient.MIGRATED_TO_NEW_20260310/src/main/java/dbclient/sync/drivers/LtcDriver.java`**

### 具体修改

#### 1. 新增实例变量
```java
private volatile long anchorSampleTime = 0;      // 锚点样本索引
private volatile double anchorSeconds = 0.0;     // 锚点时间线秒数
private volatile long samplesWritten = 0;        // 累计输出样本数
private volatile double bmcPhase = 0.0;          // BMC调制相位（确保连续）
private final double THRESHOLD_FRAMES = 2.0;     // 重锚阈值（2帧）
```

#### 2. start() 方法修改
```java
// 在 line.start() 后重置锚点状态
anchorSampleTime = 0;
anchorSeconds = 0.0;
samplesWritten = 0;
bmcPhase = 0.0;
```

#### 3. update() 方法重写
- 计算预期时间：`expectedSeconds = anchorSeconds + (samplesWritten - anchorSampleTime) / sampleRate`
- 计算偏差：`deviation = |newSeconds - expectedSeconds|`
- 仅在 `deviation > frameDuration * THRESHOLD_FRAMES` 时重锚
- 更新锚点：`anchorSeconds = newSeconds; anchorSampleTime = samplesWritten;`

#### 4. pumpAudio() 方法重构
- **时间计算**：改用锚点样本计数
  ```java
  long sampleIndex = samplesWritten + i;
  double t = anchorSeconds + (sampleIndex - anchorSampleTime) / (double) sampleRate;
  ```
- **BMC相位连续**：`bmcPhase` 作为实例变量持续累加，不重置
- **样本计数**：每个 buffer 后 `samplesWritten += bufferSamples`

### 核心机制

#### 1. 帧锁定步进
- 放弃实时 `seconds` 直接驱动，改为样本计数驱动
- 时间线 `t` 由锚点 `(anchorSeconds, anchorSampleTime)` 和当前样本偏移计算
- 消除了 wall-clock 时钟抖动导致的帧边界不稳定

#### 2. 重锚阈值
- 默认阈值：2帧时间（80ms @ 25fps）
- 仅在时间线跳变（如 CDJ Hot Cue、曲目切换）或累计漂移超阈值时重锚
- 减少边界重贴导致的帧时基抖动

#### 3. BMC相位连续
- `bmcPhase` 实例变量，不因帧边界变化重置
- 相位增量连续累加，确保调制信号相位连续性
- 但重锚时相位未调整，可能与新帧边界不完全匹配（已知风险）

### 预期效果
1. **帧连续性改善**：帧边界应稳定，无频繁重贴
2. **Resolume 锁码**：消除 `-0/0` 闪烁，实现稳定锁码
3. **信号质量**：保持现有信号电平（gainDb=-16dB），波形应更稳定

2. **简化编码**：当前仍是双相位简化版，非标准 80-bit LTC + BMC

## 02:56 标准 LTC 编码实现完成

### 实施概述
基于用户明确的指令要求，实施了**最小可锁码版本**的标准 LTC 编码，仅支持 25fps non-drop，保持最小修改范围。

### 技术实现

#### 1. 新增内部类结构
在 `LtcDriver.java` 中新增了两个内部类：

**① LtcFrameBuilder（标准 80-bit 帧构建器）**
- 完全按照 SMPTE 12M-1 标准实现
- 同步字：`0x3FFD`（0011 1111 1111 1101）
- 包含所有标准字段：
  - 帧数（4 bits + 十位 4 bits）
  - 秒数（个位 4 bits + 十位 4 bits）
  - 分钟数（个位 4 bits + 十位 4 bits）
  - 小时数（个位 4 bits + 十位 4 bits）
  - 用户位（32 bits，全0）
  - 保留位（11 bits，全0）
- 位序：LSB 先传（bit 0 最先传输）

**② LtcBmcModulator（标准 BMC 调制器）**
- 实现标准双相标记编码规则：
  - 每个 bit 起始必翻转电平
  - bit=1 时，半 bit 位置再翻转一次
  - 保持电平连续性，避免帧边界相位重置
- 状态机管理：`currentLevel`, `lastBit`, `halfBitFlip`, `sampleCounter`

#### 2. 保留的核心机制
- **锚点样本计数时基**：`anchorSampleTime`, `anchorSeconds`, `samplesWritten`
- **重锚阈值**：`THRESHOLD_FRAMES = 2.0`（偏差 > 2 帧才重锚）
- **帧锁定步进**：基于样本计数驱动，避免 wall-clock 抖动

#### 3. 编码流水线重构
`pumpAudio()` 方法完全重构为标准流水线：
```
锚点样本计数 → 时间计算 → 80-bit帧构建 → BMC调制 → 音频输出
```

**具体流程**：
1. 基于锚点计算当前时间：`totalSec = anchorSeconds + (sampleIndex - anchorSampleTime) / sampleRate`
2. 转换为时/分/秒/帧：
   - `hour = (totalSeconds / 3600) % 24`
   - `minute = (totalSeconds % 3600) / 60`
   - `second = totalSeconds % 60`
   - `frame = (int)((totalSec - Math.floor(totalSec)) * fps)`
3. 构建标准 80-bit LTC 帧：`LtcFrameBuilder.buildFrame(hour, minute, second, frame, false)`
4. BMC 调制生成音频样本：`modulator.nextSample() * amp`
5. 16-bit PCM 编码输出

#### 4. 最小版本支持
- **帧率**：25 fps（固定）
- **模式**：non-drop（`dropFrame = false`）
- **目标**：先实现可锁码版本，再扩展多帧率支持
- **扩展性**：保留结构，但当前仅实现 25fps

## 03:30 致命差异点定位：帧推进逻辑错误 + 偶校验位缺失

### 对照分析结果
**对照样本**: `/home/shenlei/下载/LTC_00000000_5mins_25fps_48000x16.wav`  
**对照内容**: 首帧及前 200ms (约 5 帧)

### 首个明确差异点

**【致命差异 1】帧推进逻辑 - 时间码不前进!**

**位置**: `LtcBmcModulator.nextSample()`  
**问题代码**:
```java
if (currentBitIndex >= 80) {
    currentBitIndex = 0;  // ❌ 重置为0，重复同一帧!
}
```

**影响**: 
- 同一帧内容无限循环发送
- 时间码永远停留在 `00:00:00:00`
- Resolume 检测到恒定时间码，无法锁定

**【致命差异 2】bit 67 偶校验位缺失**

**位置**: `LtcFrameBuilder.buildFrame()`  
**问题**: bit 67 强制为 `false` (0)，未计算偶校验  
**标准**: bit 67 应为 bits 0-63 的偶校验位  
**影响**: 部分接收器会因校验位错误拒绝帧
