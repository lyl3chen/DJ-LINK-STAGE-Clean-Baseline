# DJ LINK STAGE 项目总体架构说明

## 1. 项目定位

本项目不是单一的 DJ Link 同步工具，而是一个面向演出场景的播放状态中枢、同步输出平台与事件触发平台。

项目当前唯一真实输入源是 Pioneer CDJ/XDJ/Mixer，经 beat-link 获取播放状态、时间、BPM、Pitch、曲目信息。后续将加入本地播放模式，使系统在无 CDJ 的情况下，也能通过本地导入音乐文件，分析音频并输出统一数据。项目未来还将加入事件触发功能，用于根据播放行为与音乐结构驱动外部系统。

项目长期目标是形成一个"输入源无关"的统一平台，不管底层来源是 DJ Link 设备还是本地播放器，上层都能统一获得播放状态、时间、节拍、曲目信息，并分别提供连续同步输出与事件触发输出。

---

## 2. 当前与未来的总体架构

项目建议长期按 6 层架构组织：

### 2.1 输入源层

负责"数据从哪里来"。

**当前输入源**：
- DJ Link Source
  - CDJ / XDJ / Mixer
  - beat-link
  - 设备真实播放状态

**后续新增**：
- Local Playback Source
  - 本地音乐文件导入
  - 本地音频解码与播放
  - BPM / beat grid / waveform / 段落分析

这两类输入源未来都应统一抽象为同一种上层结构，而不是让上层逻辑绑定某一种来源。

### 2.2 统一播放状态核心

负责统一管理当前有效输入源，并对外提供唯一播放状态真源。

应统一输出的信息包括：
- 当前活动输入源
- 播放 / 暂停 / 停止状态
- 当前曲目时间
- BPM
- Beat / Phase
- Cue / 跳点
- Track 信息
- 段落 / 结构信息（后续）

这里是整个系统的中枢，后续连续同步层和事件触发层都应从这里取数据。

### 2.3 连续同步核心

负责把统一播放状态转换成连续型同步输出。

**当前已存在或将保留的连续同步方向**：
- Ableton Link
- Titan BPM
- MA2 BPM

**后续会重新加入**：
- LTC
- MTC

这一层只负责"连续同步输出"，例如：
- 当前 BPM
- 当前时间线
- 当前节拍/相位

不应混入离散事件触发逻辑。

### 2.4 事件触发核心

负责检测离散事件并触发外部动作。

**未来事件包括但不限于**：
- 播放开始
- 暂停
- 停止
- 恢复播放
- Hot Cue 跳转
- 快进 / Seek
- 切歌
- Drop / Break / Chorus / Build-up 等段落事件
- 自定义规则事件

这一层与连续同步层必须分离。同步层负责"持续告诉外部现在走到哪"，触发层负责"告诉外部刚刚发生了什么事件"。

### 2.5 输出适配层

负责把连续同步和事件触发输出到外部系统。

**当前已有输出**：
- Ableton Link
- Titan API
- MA2 Telnet
- Console API

**后续新增**：
- LTC
- MTC
- 事件触发输出（可能包括 OSC、MIDI、HTTP、WebSocket 等）

这一层建议拆分为：
- 连续同步输出驱动
- 事件触发输出驱动

不要把两类能力混到一个 driver 里。

### 2.6 表现与控制层

负责展示、配置与控制。

**当前包括**：
- WebUI
- Dashboard
- I/O Setup
- API
- WebSocket / Event 推送

**后续除了展示设备状态与输出状态，还应展示**：
- 当前输入源类型（DJ Link / 本地）
- 当前曲目分析信息
- 当前触发规则状态
- 当前事件日志

## 3. 当前项目中的核心模块

### 3.1 设备输入与协议层

| 模块 | 职责 |
|------|------|
| DeviceManager | DJ Link 设备管理 |
| DbConnection | 数据库连接 |
| DbProtocol | 协议解析 |
| MetadataParser | 元数据解析 |
| SystemState | 系统状态管理 |

**职责范围**：
- 读取 DJ Link / beat-link 设备状态
- 维护播放器在线状态、曲目信息与播放状态
- 不负责外部输出

### 3.2 同步输出框架

| 模块 | 职责 |
|------|------|
| SyncOutputManager | 同步输出管理器 |
| OutputDriver | 输出驱动接口 |

**职责范围**：
- 统一管理输出驱动生命周期
- 从统一状态核心获取数据
- 分发给各类输出模块

### 3.3 当前保留的输出模块

- AbletonLinkDriver： Ableton Link 同步输出
- TitanApiDriver / TitanAdapter： Titan API 同步输出
- Ma2TelnetClient / Ma2BpmDriver： MA2 Telnet 同步输出
- ConsoleApiDriver： Console API 同步输出

### 3.4 表现与控制层

| 模块 | 职责 |
|------|------|
| WebServer | Web 服务器 |
| JettyServer | Jetty 服务器 |
| EventPusher | 事件推送 |
| WebUI | 前端界面 |

**职责范围**：
- HTTP API 提供
- WebSocket 事件推送
- 前端页面展示与配置

## 4. 时间码模块的定位

当前仓库中，旧 LTC / MTC 及其相关残留已经被清理。后续 LTC / MTC 必须从 0 重做。

### 4.1 正确主线

```
CDJ 事件层 / 本地播放事件层
    ↓
统一播放状态核心
    ↓
本地稳定时间流
    ↓
LTC / MTC 编码输出层
```

### 4.2 核心原则

- LTC 与 MTC 必须共用同一时间源核心
- 输出层只负责编码和发送
- 不得再让 LTC / MTC 各自维护独立时间推进逻辑
- 必须使用本地单调时钟或等价稳定样本/帧时基
- 不得回到旧的补丁式 wall-clock 重贴路线

## 6. 后续事件触发系统的位置

事件触发系统不应塞进时间码模块，也不应塞进某个单一输出驱动。

### 正确位置

```
统一播放状态核心
    ↓
事件检测器
    ↓
事件触发输出层
```

### 触发事件示例

- 播放开始 → 事件触发
- 暂停 → 事件触发
- 停止 → 事件触发
- Hot Cue → 事件触发
- 切歌 → 事件触发
- 段落切换 → 事件触发

然后再由外部适配层把这些事件送给 Titan、MA2、未来的 AI 规则引擎或其他系统。

---

## 7. 最关键的架构原则

### 7.1 输入源无关

上层逻辑不应绑定"CDJ 时间"、"CDJ BPM"，而应统一为 sourceTime、sourceBpm、activeSource 等通用结构。

### 7.2 一个播放状态真源

所有同步输出与事件触发都必须从统一播放状态核心读取。

### 7.3 连续同步与事件触发分离

- 同步层：负责"持续前进"
- 触发层：负责"发生了什么"

### 7.4 时间码必须共用同一时间线

LTC 与 MTC 后续必须共用同一个 Timeline 核心。

### 7.5 后续时间码从 0 重做

不继承旧补丁、旧状态机、旧 legacy 路线。

## 9. 总结

DJ LINK STAGE 的长期正确形态应当是：

一个同时支持外部 DJ 设备与本地播放器输入、同时支持连续同步输出与事件触发输出、并以统一播放状态为核心的演出控制平台。

### 当前阶段

- 输入源：主要是 DJ Link / beat-link
- 输出能力：保留了 Ableton Link、Titan、MA2、Console API
- 时间码模块：已清空，准备从 0 正确重建

### 下一阶段

- 重建 LTC / MTC
- 加入本地播放模式
- 加入事件触发系统
